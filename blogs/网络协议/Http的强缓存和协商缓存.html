    Http的强缓存和协商缓存

    CoderTao 2024/3/28 Http缓存

    # 强缓存

    客户端直接从本地缓存中获取资源，而无需与服务器进行通信。强缓存的判断是通过缓存头部字段来完成的，例如设置了合适的Cache-Control和Expires字段。

    # 1. Expires 强缓存

    当客户端请求资源时，获取本地时间戳跟设置的缓存过期时间对比，未过期，走强缓存；已过期，重新发起请求。

    const express = require("express");
const cors = require("cors");
const app = express();

app.use(cors());

app.get("/expires", (req, res) => {
    // 禁用缓存策略
    res.setHeader("Cache-Control", "no-store");
    //设置过期时间
    res.setHeader("Expires", new Date("2024-04-25 14:23:00").toUTCString()); 
    res.json({
        msg: "expires cache",
    });
});

app.listen(9000, () => {
    console.log("server is running at http://localhost:9000");
});

    # 2. Cache-Control 强缓存

    • no-store 禁用缓存策略
    • no-cache 禁用强缓存，使用协商缓存 ( Last-Modified 或 Etag )
    • public 浏览器、代理服务器（CDN）等均能缓存
    • private 仅客户端能缓存
    • max-age 资源缓存时长（单位秒）
    const express = require("express");
const cors = require("cors");
const app = express();

app.use(cors());

app.get("/cache-control", (req, res) => {
    // 设置缓存时间为5秒
    res.setHeader("Cache-Control", "public, max-age=5");
    res.json({
        msg: "cache-control",
    });
});

app.listen(9000, () => {
    console.log("server is running at http://localhost:9000");
});

    区别：

    1. Expires 是HTTP/1.0的产物，Cache-Control 是HTTP/1.1的产物
    2. 两者同时存在，优先级：Cache-Control > Expires
    3. Expires 是控制时间点， Cache-Control 是控制时间段；若本地时间不准确，会导致 Expires 的缓存失效。

    # 协商缓存

    客户端会发送带有缓存数据标识的请求头部字段，以向服务器验证资源的有效性。强缓存未命中的情况有：max-age 过期或服务器响应中设置Cache-Control: no-cache

    # 1. Etag 和 If-None-Match 协商缓存

    Etag 是对该资源的唯一标识，在上一次请求的response header中，在下一次向服务器发送请求时，会将上一次的 Etag 值放到request header里的 If-None-Match 里，服务器接收到 If-None-Match 的值后，跟该资源文件的 Etag 值做比较，如果相同，则表示资源文件没有发生改变，命中协商缓存。当资源有变化，Etag 就会重新生成。

    const express = require("express");
const cors = require("cors");
const crypto = require("crypto");
const fs = require("fs");
const app = express();

app.use(
    cors({
        exposedHeaders: ["If-None-Match", "Etag", "If-Modified-Since", "Last-Modified"],
    })
);

app.get("/etag", (req, res) => {
    // 设置缓存时间为10秒并走协商缓存
    res.setHeader("Cache-Control", "no-cache, max-age=10");
    // 获取请求头中的If-None-Match
    const IfNoneMatch = req.get("If-None-Match");
    // 生成 ETag
    const etag = crypto.createHash("sha1").update(fs.readFileSync("./src/index.js")).digest("hex");
    // 设置 ETag
    res.setHeader("ETag", etag);
    res.json({
        data: { IfNoneMatch, etag },
        msg: "etag",
    });
});

app.listen(9000, () => {
    console.log("server is running at http://localhost:9000");
});

    # 2. Last-Modified 和 If-Modified-Since 协商缓存

    Last-Modified 和 If-Modified-Since：服务器通过 Last-Modified 响应头告知客户端资源的最后修改时间。客户端在后续请求中通过 If-Modified-Since 请求头携带该时间，服务器判断资源是否有更新。无更新命中缓存，返回 304 状态码；否则，重新请求服务器，返回 200 状态码。

    const express = require("express");
const cors = require("cors");
const fs = require("fs");
const app = express();

app.use(
    cors({
        exposedHeaders: ["If-None-Match", "Etag", "If-Modified-Since", "Last-Modified"],
    })
);

app.get("/last-modified", (req, res) => {
    // 设置缓存时间为10秒并走协商缓存
    res.setHeader("Cache-Control", "no-cache, max-age=10");
    // 获取请求头中的If-Modified-Since
    const IfModifiedSince = req.get("If-Modified-Since");
    // 获取文件最后修改时间
    const lastModified = fs.statSync("./src/index.js").mtime.toISOString();
    // 设置 Last-Modified
    res.setHeader("Last-Modified", lastModified);

    res.json({
        data: {
            IfModifiedSince,
            lastModified,
            status: IfModifiedSince === lastModified,
        },
        msg: "last-modified",
    });
});

app.listen(9000, () => {
    console.log("server is running at http://localhost:9000");
});

    区别：

    1. 方式：Etag 是对资源的一种唯一标识，而 Last-Modified 是该资源文件最后一次更改时间。
    2. 精确度：Etag > Last-Modified
    3. 性能：Etag < Last-Modified，Last-Modified 只需要记录时间，而 Etag 需要服务器通过算法来计算出一个hash值。
    4. 优先级：Etag > Last-Modified
